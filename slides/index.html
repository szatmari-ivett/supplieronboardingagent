Supplier Onboarding Agent

ALDI Developer Challenge prototype

Conversational onboarding across procurement, ERP, and cloud apps

Problem

  • Onboarding takes 6.5–10 weeks
  • Status is fragmented across ERP, procurement, and cloud tools
  • Manual coordination slows time-to-market

Vision

  • Guide users end-to-end in one conversational flow
  • Detect duplicate suppliers before creating records
  • Provide aggregated status on demand

High-level architecture

Conversational UI
    -> FastAPI POST /chat (SSE progress + message events)
    -> LangGraph orchestrator
         router -> act? -> respond
    -> Tools (duplicate, onboarding, status)
    -> Resilient connectors
    -> Mock ERP / Procurement / Cloud
Memory: SQLite process store + LangGraph checkpointer

Why LangGraph?

  • Explicit router / act / respond flow with conditional routing
  • Help intents skip tool execution without extra branching in the UI
  • Session memory via checkpointer; stream mode for live progress events

Rejected: ad-hoc prompt loops (hard to observe), heavy multi-agent frameworks (too much ceremony for a prototype)

LLM choice

  • Default: MockLLM for deterministic demos and offline review
  • Optional: OpenAI GPT-4o mini or Anthropic Haiku via function calling
  • Fallback: MockLLM when API keys are absent or provider calls fail at startup

Trade-offs: capability vs cost vs latency vs data residency

Agent memory

  • Short-term: conversation thread in LangGraph memory
  • Long-term: onboarding process state in SQLite
  • Retention: prototype-local SQLite files, replaceable with managed store in production

Tools and MCP mapping

  • duplicate_check -> supplier master lookup service
  • create_onboarding -> procurement package + ERP + cloud workflow
  • aggregate_status -> cross-system status API

Prototype uses Python service functions; production mapping: MCP servers per platform with OAuth/service accounts

Note: MCP is not implemented here because the challenge expects mocked backends; in production, dedicated MCP servers per platform (procurement, ERP, cloud) would expose scoped tools over OAuth/service accounts.

Duplicate detection

  • Normalize VAT, DUNS, name, address
  • Exact match on VAT/DUNS
  • Fuzzy match on name + address via RapidFuzz
  • Thresholds: 95+ auto, 80–95 review, below 80 new supplier

Cross-system status

  • Canonical phases: initiated -> duplicate_check -> package_created -> erp_sync -> cloud_compliance -> active
  • Map subsystem states into canonical phases
  • Health: healthy / degraded / stale

Async, retry, idempotency

  • Task states: pending, in_progress, waiting_external, done, failed
  • Exponential backoff retry on transient connector failures
  • Idempotency keys on create operations
  • Fault injection flags for demo and testing

Security and privacy

  • Secrets only in environment variables
  • Guardrail example: VAT IDs and emails masked in structured logs before write
  • Guardrail example: chat input validated (length limits, safe session_id pattern)
  • Production would add RBAC, audit trails, and per-system scoped tokens

Demo walkthrough

  1. Duplicate check for FreshFarm GmbH
  2. Start onboarding for NewOrganic Ltd
  3. Ask for ONB-001 status
  4. Enable ERP fault to show degraded health

Alternatives considered

  • Single prompt chain -> rejected: weak tool control and observability
  • Direct UI over APIs -> rejected: no conversational guidance
  • Real integrations in prototype -> rejected: violates challenge scope
